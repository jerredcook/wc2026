The verdict

You've built something genuinely good. For a single 4,680-line HTML file with no framework and no build step, the engineering is well above hobby baseline: every score lives in exactly one place and all eleven sections derive from it, so they can never disagree. The 2026 tiebreaker logic is faithful down to recursive head-to-head before goal difference. There's a real modal focus trap, keyboard activation, reduced-motion support, and a three-tier data fallback that degrades to a self-contained archive instead of a blank page.

The single-file, no-build decision is still the right call — it's what keeps deployment a git-push and lets the app run from a file on disk.

But the review found real bugs, and a cluster of them were time-critical — one was corrupting data you intend to keep forever. Seven of those have now been fixed and shipped. What remains splits cleanly into three jobs: protect the permanent archive before July 20, sharpen the live experience for the knockout run, and a longer tail of cleanup for after the Final. This document lays out all of it — what's strong, every confirmed defect with its exact location, and a prioritized roadmap.

Resolved this session

Seven fixes already shipped

The urgent batch — the two deadline-bound risks (a penalty-shootout Final with no champion; group tables corrupted by a rematch) plus the biggest live-experience and knockout gaps. Committed and verified against live ESPN data.

Penalty shootouts are now first-class, end to end

was critical ×3Shipped

The parser read only the 90-minute score, so a drawn knockout tie showed no winner (a shootout Final could never crown a champion) and shootout kicks were counted as goals — Netherlands 1–1 Morocco carried seven "goals" into the Golden Boot. Now the shootout tally and winner flag are parsed (in both the app and the snapshot script), goal/timeline events are capped at the real score, and the tally + gold-highlighted winner render across the bracket, rows, popover and champion line. The archive was re-captured clean.

index.html parseESPNEvents · snapshot-live.mjs · bkChampHtml / koWinner
Both R32 shootouts now recorded correctly; zero phantom goals in the snapshot.

A knockout rematch can no longer corrupt the group tables

was highShipped

Live results were matched to fixtures by team pair in either order, with no round or date check. From the quarterfinals on, two teams that met in the group stage can meet again — and the knockout score would have overwritten the completed group fixture, silently rewriting the final tables. Now live scores apply only to group fixtures; knockout scores flow through the bracket.

was mediumShipped

Auto-refresh only armed while a match was already live, so opening the app at 2:45 for a 3:00 kickoff left it frozen forever. A one-shot timer now fires at the next scheduled kickoff (re-arming across rest days), and the page refreshes when the tab regains focus.

Before July 20 · highest stakes

Protecting the permanent archive

Your stated goal is that this site outlives ESPN's feed and stays a complete record. The snapshot as captured today is not yet that record — these are the gaps between "it works now" and "it's safe forever."

The snapshot's safety guard only counts events — it can clobber good data

HighOpen

The overwrite guard compares only events.length. Match summaries (goals, lineups, commentary) aren't guarded at all — a run where the summary endpoint hiccups passes the guard and can overwrite good archived detail with less. Fix: union-merge events by id, and never let a summary shrink.

scripts/snapshot-live.mjs · 226–231

The daily capture Action can fail silently, forever

HighOpen

Every failure mode is a green run: the fetch helper swallows errors and returns null, an empty parse yields [], and the shrink-guard exits 0. If ESPN changes shape or blocks the runner mid-tournament, the workflow keeps "succeeding" daily with no commit and no alert. Make the guard exit 1, and add a staleness check that fails when the archive stops advancing.

scripts/snapshot-live.mjs · getJSON 42–52 · snapshot.yml

Snapshot fallback is all-or-nothing

MediumOpen

The committed snapshot is only consulted when the live path returns zero results. The likely post-tournament failure is ESPN returning partial data (dropping old dates) with a non-empty length — which passes the check, so the archive is never consulted and users see an incomplete tournament. Merge the snapshot under live data, keyed by match id.

index.html · refreshScores 4548 / 4554

force-cache can pin users to a stale, pre-Final snapshot

MediumOpen

Both snapshot fetches use cache:'force-cache', which returns any cached copy without revalidating. A browser that cached the snapshot on July 10 could, a year later when ESPN is gone, show that stale mid-tournament copy indefinitely. Switch to default so it revalidates against the ETag.

index.html · 2195 (getWCSnapshot) · 4556 (refreshScores)

The snapshot drops upcoming fixtures — pure-archive mode loses future pairings

MediumOpen

The capture keeps only FT/LIVE events, but the live app deliberately reads the full window because ESPN already knows the upcoming knockout matchups. In snapshot-only mode, the bracket loses those known-but-unplayed pairings, and a drawn match's advancer is unknowable until its next game is captured.

scripts/snapshot-live.mjs · 199

2026 is never folded into the historical archive

MediumOpen · post-final

The tournaments table ends at 2022, and the head-to-head index is built only from the archived year files. Unless a 2026 entry is added and the two build scripts re-run after the Final, the History section and the popover H2H will permanently exclude this World Cup — even though the live snapshot has richer data than the historical files.

scripts/build-history.mjs 30–53 · build-h2h.mjs

~70 lines of feed parsing are hand-duplicated app ↔ snapshot

MediumOpen

The snapshot script's parser is a manual port of the app's, and the h2h builder re-implements the app's name normalizer with a "MUST match" comment. Any future edit to one and not the other silently corrupts the archive. A tiny shared module (one <script src>, no build step) would end the drift risk — and make this logic unit-testable.

snapshot-live.mjs 80–152 · build-h2h.mjs 16–22

The next 18 days · live experience

Sharpening the knockout run

Small, high-leverage changes for a dad-and-son checking scores daily through to the Final — most reuse data and helpers the app already has.

~39 ESPN requests every 60 seconds during live play

MediumOpen

Every refresh re-fetches all 39 tournament days, most with no-store — roughly 2,300 requests an hour on a phone, nearly all of them immutable finished days. Poll only today ±1 on the live loop; keep the full-window pull for initial load and the manual Refresh. Better battery, less rate-limit exposure.

index.html · fetchScoresFromESPN 4509–4536

The group-stage UI still owns the default layout

MediumOpen

Groups ended June 27, yet "Group Standings — Live" sits open in slot three while Results and the Golden Boot — the things that actually change daily now — start collapsed. A one-time "knockout phase" flag (like the existing layout migration) should demote the frozen tables and promote the live sections.

The bracket opens scrolled to the dead left edge on phones

MediumOpen

The bracket is 1,260px wide inside a scroller, so a ~390px phone opens showing only the left Round-of-32 column — matches that are all finished after July 3 — with the trophy and the live rounds two screens to the right, hinted only by buried prose. Auto-scroll to the current round after render.

"Tap any match for detail" usually opens a team profile

MediumOpen

Team-name links are given priority over the enclosing match row and fill most of a short row's width, so a tap usually lands on the team card instead of the match card the subtitle promises. Worth rebalancing the hit areas — also the son-friendliness fix.

Rest days dead-end the hero section

MediumOpen

Five of the remaining days have no fixtures, and on those days the Today bar — the page hero — says only "No World Cup matches scheduled today." Show the next one instead: "Next: Quarterfinals — Thu Jul 9, 2:00 PM." The kickoff instant is already computed.

Offline mode shows no scores, despite claiming "saved scores"

MediumOpen

The service worker precaches only the shell — the snapshot is never cached, and is fetched only when the live path fully fails, which never happens online. So the installed app offline shows an empty tournament while the status line promises otherwise. Precache the snapshot and fall to it when offline.

Cleanup worth doing

ESPN-sourced team names are rendered unescaped in places

MediumOpen

The escaping helper is used diligently in most render paths, but feed-derived names flow raw into the bracket and knockout rows. Escaping at the two funnels every name passes through — teamLink and bkTeamRow — closes the whole class in about six edits, with no visual change.

index.html · teamLink 1682 · bkTeamRow 3248

A dead call to api.anthropic.com delays the real fallback

MediumOpen

A leftover from the artifact era posts to the Anthropic API with no key — from GitHub Pages it always fails, adding a guaranteed-failing round-trip before the snapshot fallback, and its user-facing message misleads. Pure deletion: ~25 lines gone, one failure mode removed.

index.html · fetchLiveScores 4330–4348

"Road to the Final" traces a semifinal loser into the Final

LowOpen

The path tracer picks a team's deepest appearance and, on a tie between the semifinal and third-place slots, keeps the earlier one — so a losing semifinalist's traced road runs into the Final rather than the third-place match.

index.html · traceBracketPath 3328–3331

The README documents a flag that doesn't exist

LowOpen

The docs describe --full and a "capture up to today" default; the script actually defaults to the full window and takes --today. Harmless today, but exactly the kind of thing that will mislead you at the one moment it matters — the manual post-Final capture.

Minor notes

The smaller stuff

Low-severity observations — worth a pass, none urgent.

App identity is still "Stadiums." The title tag and H1 say "All 16 Stadiums" — it stopped being a stadiums page a month ago.

Developer jargon leaks into user copy — "Claude's view," "served over http(s)," "file://" in status and empty-state text.

Korea DPR maps to South Korea in the alias table. Latent (North Korea isn't in 2026), but wrong if the historical archive ever routes through it.

Sub-9px type & low-contrast metadata on bracket cards fall below comfortable legibility.

The follow bar hides kickoff times for knockout matches even though the time is now known.

The snapshot workflow has no pull/rebase before push — a race with your own push can silently lose a day's capture.

The service worker caches error responses (no res.ok check), which can poison the offline shell mid-deploy.

manifest.webmanifest is 55KB of inlined base64 icons — and Bubblewrap can't consume data-URI icons, blocking a Play Store TWA.

Toggle-pill CSS is copy-pasted ten times — every style tweak is a 10–15 edit tax the section registry could remove.

Parked Pick'em still runs in the hot path — invoked on every refresh though it early-returns.

Wholesale 60s innerHTML rebuilds drop keyboard focus during live matches.

No .gitignore — .claude/ sits untracked-but-unignored in every status.

Credit where due

What's genuinely strong

Forty strengths were catalogued across the six lenses. The ones that matter most:

✓ One source of truth

Every fixture lives in one place; all eleven sections are pure recomputations, so scores can never disagree between them.

✓ Faithful football rules

The 2026 tiebreakers are implemented down to recursive head-to-head before goal difference, with an honest "disciplinary points not modelled" caveat.

✓ Degrades to an archive

A three-tier fallback ends at a committed snapshot with an honest "Archived" status line — the app becomes a self-contained record, not a blank page.

✓ Real accessibility

A MutationObserver focus trap with a nesting stack, Enter/Space activation on every row, focus-visible outlines, reduced-motion support — unusual for a hobby single-file app.

✓ Thoughtful time zones

Local-zone default with a one-time migration, an ET/local/picked toggle, and midnight-crossing games shown in gold with a "→ next-day" tag.

✓ Update rollout is solved

The service worker is network-first with skipWaiting and cache cleanup, correctly scoped to same-origin — a git-push reaches online users on next load.

✓ Bracket craft

Plain-language slot labels, projected-placement tags with tooltips, a "road to the final" tracer, and connectors that light up as results land.

✓ A greppable 4,680 lines

A consistent two-letter prefix convention across CSS, ids and functions, with banner comments — what keeps a single file navigable without modules.

How this was made

Method

Six specialist reviewers read the whole codebase in parallel, each through one lens — architecture, correctness, product & UX, data pipeline, PWA & performance, and testing & sustainability. Every high- or medium-severity claim was then handed to a separate, independent verifier whose only instruction was to refute it against the real code and data, defaulting to "not a real bug" when it couldn't reproduce the reasoning.

Thirty-one findings went through that gauntlet. All thirty-one survived; none were refuted. That's why this document states them as fact rather than suspicion — each one was reproduced from a specific line of code or a specific row in the committed snapshot. The seven marked Shipped were fixed and verified against live ESPN data in the same session.
